# "그룹개발" 방법론

> 이 프로젝트(agent-dashboard)의 개발 워크플로우 표준. 결정 근거: [`decision.md` D20](../decision/decision.md) (재발 방지 철학은 D13 인시던트와 연결).

## 정의

**그룹개발**은 한 명의 오케스트레이터(메인 세션)가 개발을 여러 **하위 에이전트**에 분담하고, **각 산출물을 전담 검증 에이전트가 판단**하게 하는 개발 방식이다. 사람이 매번 수동으로 하던 "스펙대로 짰나 / 전체가 도나" 검증을, 도구 권한이 제한된 검증 에이전트로 **표준화**한 것이 핵심이다. "혼자 다 짜고 혼자 확인"이 아니라, **개발자(다수) + 검증자(매칭)** 가 한 팀처럼 움직인다.

## 4단계 흐름

1. **계획** — 사용자와 함께 무엇을 만들지, 어떤 모듈로 나눌지, 의존 관계는 무엇인지 정한다(큰 결정은 `decision.md`에 기록).
2. **위임(개발)** — 개발을 하위 에이전트에 분담한다.
   - **독립 작업 = 병렬**(예: 순수 함수 모듈 여러 개를 동시에).
   - **의존 작업 = 순차**(앞 산출물이 있어야 다음이 가능할 때).
   - 모든 개발 하위 에이전트 프롬프트에는 **보안 클로즈**를 코드 가드와 함께 명시한다(아래).
3. **dev-verifier 검증(매칭)** — **개발 하위 에이전트 1개마다 매칭되는 검증 에이전트**가 그 산출물이 배정된 목적(스펙)·범위·보안 클로즈대로인지 **독립적·적대적으로 판단**한다. PASS/FAIL과 (FAIL 시) 필요한 수정만 보고하고 **직접 고치지 않는다**.
4. **project-verifier 검증(종합)** — 모든 작업이 끝난 뒤 **프로젝트 전체가 정상 동작**하는지 종합 점검한다(빌드·CLI·HTTP·통합·보안). 마지막에 한 번.

```
[계획] → [개발 위임: A1‖A2‖A3 → (의존) B → C]
            │        │        │       │    │
          dev-v    dev-v    dev-v   dev-v dev-v      ← ③ 각 산출물마다 매칭 검증
                          ⋯ 모든 작업 종료 ⋯
                         → [project-verifier]        ← ④ 전체 동작 종합 판단
```

## 두 검증 에이전트의 역할과 도구 제한

| | dev-verifier | project-verifier |
|---|---|---|
| 단계 | ③ (개발 에이전트마다 매칭) | ④ (전체 종료 후 1회) |
| 판단 범위 | 그 에이전트의 **배정 스펙·범위** | **프로젝트 전체** 동작 |
| 입력 | (a) 원래 과제/스펙, (b) 배정 파일 목록, (c) 산출물 보고 | (전체 코드베이스) |
| 검사 | 범위 준수·스펙 충족·보안 클로즈·회귀 | 빌드/구문·CLI 스모크·HTTP 스모크·통합/회귀·보안 §4 |
| 출력 | `verdict` + 항목별 근거 + `required_fixes` | `verdict` + 항목별 근거 + `breakage` + `notes` |

**도구 제한(코드 가드) — 가장 중요한 설계:**
두 에이전트 모두 정의 파일(`.claude/agents/*.md`)의 frontmatter에서 `tools: Read, Grep, Glob, Bash` 만 가진다. **Edit / Write / NotebookEdit이 없다.** "판단만 하고 코드는 고치지 않는다"를 프롬프트 지시가 아니라 **도구 권한으로 강제**한 것이다(D13 인시던트 교훈 — 보안은 프롬프트가 아니라 코드로 강제, [`enforce-security-with-code-guards`] 철학과 일관). 검증자가 코드를 만질 수 없으므로, 검증과 개발이 구조적으로 분리된다.

**하위 에이전트 모델 고정 (2026-07-02 결정):**
그룹개발의 **모든 하위 에이전트**(개발·조사·검증 전부)는 **Opus 4.8 + 높은 추론(`effort: high`)** 으로 돌린다.
메인 세션이 어떤 모델로 돌든, 위임된 작업의 품질을 모델 선택 변수에서 분리하기 위함.
- 검증자: `.claude/agents/dev-verifier.md` · `project-verifier.md` frontmatter에 `model: claude-opus-4-8` + `effort: high` 고정.
- 개발자: `.claude/agents/dev-worker.md`(동일 고정)를 사용하거나, 임시 유형(general-purpose 등)으로 띄울 때는 호출 시 모델을 opus로 명시.

## 호출 방법

- **dev-verifier**: 개발 하위 에이전트 하나가 작업을 끝낼 때마다, **그 작업에 매칭해** 호출한다. 호출 프롬프트에 반드시 세 가지를 넘긴다 — (a) 그 에이전트의 원래 과제/스펙/목적, (b) 수정이 허용된 배정 파일 목록, (c) 그 에이전트의 산출물 보고. 병렬로 N개를 개발했으면 dev-verifier도 N번(각각 매칭) 부른다. FAIL이면 `required_fixes`를 개발 단계로 되돌려 고친 뒤 다시 검증한다.
- **project-verifier**: 모든 개발 + 모든 dev-verifier 통과가 끝난 **마지막에 한 번** 호출한다. 입력으로 별도 스펙을 넘길 필요는 없고(전체 코드베이스를 본다), 필요하면 "이번에 추가/변경된 기능 요약"을 함께 준다.

## 보안 클로즈는 검증 항목에 포함된다

이 프로젝트는 시크릿 인접 디렉토리를 다루므로, 두 검증 에이전트는 **보안 클로즈 준수 자체를 검증 항목으로** 본다(`security-check.md` §3.5 코드 가드 · §4 자가 점검 체크리스트 반영):

- 🔒 **격리 환경 전용** — 쓰기가 일어나는 검증은 `AAD_CANONICAL`/`AAD_BACKUPS`를 임시(scratchpad) 폴더로 돌린 상태 + 임시 포트에서만. 실제 홈·실제 canonical에 쓰기 검사는 하지 않는다(D24·D37 — 더미 모드는 제거됨, 격리는 env 오버라이드로).
- 🔒 **실제 설정 파일 본문 미인용** — `~/.claude`·`~/.codex` 등 실제 파일을 읽는 것은 앱의 정상 동작이지만, 검증 보고·대화에 그 **본문·개인정보·토큰을 싣지 않는다**(이름·개수·경로만). 거부목록(`auth.json`/`oauth_creds.json`/`*.sqlite`/`sessions/`/`history.jsonl`/`.env`/`*.key`/`*.pem`)은 검증 중에도 절대 열지 않는다.
- 🔒 **쓰기 경로 단일화** — 신규 쓰기는 기존 guarded 경로(`skillPush`/`agentPush` 및 래퍼 `skillPushAll`/`agentPushAll`) 경유인지(백업·synclog 자동 상속). 우회 쓰기 경로 신설 금지.
- 🔒 **시크릿 미추적·원격 push 없음** — `git -C canonical ls-files`로 시크릿 미추적 확인, 로컬 전용 유지.

검증 에이전트의 출력(코드·판정)도 사람이 그대로 신뢰하지 않고 근거(파일:라인·명령 출력)를 보고서에 요구한다(`security.md` "AI 출력 검증" 권장과 일관).
