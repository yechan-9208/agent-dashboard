# 결정 로그 (Decision Log)

이 파일은 **프로젝트의 큰 방향을 가르는 결정**과 그 맥락을 남기는 곳이다.
새 결정이 생길 때마다 맨 아래에 추가한다. (작은 구현 세부는 여기 적지 않는다.)

기록 형식: **결정 / 맥락(왜) / 대안 / 출처·날짜**

---

## D1. 동기화 구조 = canonical 1벌 → 각 도구로 복사+변환

- **결정:** 중앙 원본(canonical) 한 벌을 두고, 각 도구(Claude/Codex/Gemini)로 **복사하면서 포맷을 변환**한다. 심볼릭 링크가 아니라 도구별 사본을 만들어 관리한다.
- **맥락:** 세 도구가 파일 위치·포맷이 달라 한 도구에서 만든 걸 다른 도구가 못 본다. 원본을 한 곳에 두면 "무엇이 어디에 있고 무엇이 다른지(drift)"를 한눈에 볼 수 있고, git으로 이력·롤백을 공짜로 얻는다.
- **대안:** 심볼릭 링크(도구별 확장 메타가 충돌, 포맷 변환 불가) → 기각.
- **출처:** chat_log (2026-06-30 이전 대화에서 확정)

## D2. 아키텍처 = CLI 코어 + 얇은 127.0.0.1 서버 + HTML 대시보드

- **결정:** 모든 로직은 **CLI(단일 소스)**. 브라우저 버튼은 **127.0.0.1에만 묶인 얇은 서버**가 받아 CLI 하위 명령을 그대로 실행만 한다(다리 역할). 서버가 안 떠 있어도 `build`로 만든 **정적 HTML 보기**는 동작한다.
- **맥락:** 브라우저는 보안 샌드박스 때문에 흩어진 로컬 파일을 직접 못 만진다. 그렇다고 무거운 백엔드를 만들 필요는 없다. "클릭-투-싱크"를 원하므로 얇은 다리만 얹고, 로직 중복은 만들지 않는다.
- **대안:** ① 서버 없는 순수 CLI(클릭 불가) ② 브라우저 File System Access API(Chromium 전용·권한 번거로움·JS에 로직 중복) → 둘 다 기각.
- **출처:** chat_log (서버리스→얇은 다리로 두 번 거쳐 확정)

## D3. 대상 = skill · agent · 지시문, 슬래시 커맨드 제외

- **결정:** 추적·동기화 대상은 **skill, agent(subagent), 지시문 파일(CLAUDE.md/AGENTS.md/GEMINI.md)** 세 가지. 슬래시 커맨드는 이번 범위에서 제외.
- **맥락:** skill은 이미 표준(agentskills.io)으로 수렴. agent는 세 도구 모두 "위임 작업자" 개념으로 수렴했으나 Codex만 TOML이라 변환 필요. 지시문은 셋 다 프로젝트 지시문이라 파일명만 바꾸면 됨.
- **출처:** chat_log

## D4. 구현 언어 = Node.js

- **결정:** **Node.js**로 구현. frontmatter=gray-matter/js-yaml, TOML=@iarna/toml, 서버=표준 `http` 모듈.
- **맥락:** 처음엔 변환(md/YAML/TOML) 편의를 이유로 Python을 골랐으나(`tomllib` 표준 내장 등), 재검토 결과 **대시보드(웹)와 백엔드를 한 언어(JS)로 통일**하는 이점을 더 중시. frontmatter는 gray-matter가 깔끔하고, TOML은 라이브러리 하나만 추가하면 됨.
- **대안:** Python(변환 편하나 화면 JS와 언어 분리) / Go(배포 강점이나 개발 속도·라이브러리 부담) → 기각.
- **출처:** 인터뷰 (2026-06-30, Python→Node로 재확정)

## D5. canonical 위치 = 프로젝트 안 `agent-dashboard/canonical/`

- **결정:** canonical을 **프로젝트 폴더 안**에 두고 자체 git 저장소로 관리. **로컬 전용**이며, 원격 push는 회사/개인 계정 확인 후 **별도 승인**해야 한다.
- **맥락:** 코드와 데이터를 한 폴더에서 관리하는 편이 단순. 경로 규칙상 `miru_github/**`는 회사 계정(1yechan)으로 커밋된다. canonical에는 지시문 등 식별정보가 들어갈 수 있어 원격 노출을 막아야 한다.
- **대안:** 홈 디렉토리 `~/ai-agent-dashboard/canonical/`(plan 원안, 코드/데이터 분리) → 기각.
- **출처:** 인터뷰 (2026-06-30)

## D6. 프로토타입 = 지시문 1트랙 수직 슬라이스부터

- **결정:** 먼저 **지시문 1트랙**으로 스캔→canonical→미리보기(diff)→적용 "한 줄기"를 끝까지 동작시킨다. 검증 후 skill·agent 트랙으로 확장.
- **맥락:** 가장 단순한 트랙(지시문=파일명만 변경)으로 전체 파이프라인(스캔·canonical·diff·백업·서버·대시보드)을 빨리 실제 동작시켜 안전 흐름을 검증하기 위함.
- **대안:** Phase 1 전체 한 번에 / View 먼저 / CLI만 먼저 → 기각.
- **출처:** 인터뷰 (2026-06-30)

## D7. skill 스캔 = 플러그인 포함 + 개인정보 확인 게이트

- **결정:** Claude skill은 플러그인 경로(`~/.claude/plugins/*/skills/`)까지 스캔 대상에 포함한다. **단, 스캔이 개인정보 관련 내용을 만나면 canonical 반영·AI 처리 전에 반드시 사용자 확인을 받는다.**
- **맥락:** 실제로 `~/.claude/skills`는 비어 있고 Claude skill은 플러그인 안에 있다(환경 탐색에서 발견). 그래서 플러그인까지 봐야 실사용 형태에 맞다. 다만 플러그인·skill 본문에 개인정보가 섞일 수 있어 보안 게이트를 건다.
- **대안:** 실재하는 것(Codex 위주)만 먼저 → 기각(개인정보 게이트를 붙이는 조건으로 플러그인 포함 채택).
- **출처:** 인터뷰 (2026-06-30)

## D8. 보안 = 화이트리스트 스캔 + 시크릿 거부목록 + Phase2 횟수만

- **결정:** 스캔은 **허용 목록(skills/·agents/·지정 지시문·플러그인 skills)만** 본다. 인증정보·DB·세션 로그는 **거부목록**으로 절대 수집·커밋하지 않는다. Phase 2의 사용량은 **호출 "횟수"만** 집계하고 로그 본문은 저장·표시·커밋하지 않는다.
- **맥락:** 스캔 대상 디렉토리 바로 옆에 `auth.json`·`oauth_creds.json`·`*.sqlite` 등 시크릿/개인정보가 있다(환경 탐색에서 확인). 실수로 canonical git에 들어가면 안 된다.
- **상세:** `docs/security/security-check.md` 참조.
- **출처:** 보안 체크 (2026-06-30)

## D9. 도구 읽기 모델 = 하이브리드 + 개발 중 더미 전용

- **결정:** 대시보드(로컬 프로그램)는 **어떤 항목이 있는지(파일명·존재)만 자동 표시**하고, **내용·drift는 사용자가 클릭한 항목만** 읽는다(PI 게이트 통과). 또한 **개발 중에는 AI(Cowork=나)가 사용자의 실제 개인 파일 내용을 읽지 않고 더미 파일로만** 만든다.
- **맥락:** 사용자가 "왜 `~/.claude/CLAUDE.md`에 접근하냐"고 우려. 두 가지가 다름 — ① AI가 내용을 외부 컨텍스트로 가져가는 것(금지 → 더미 사용), ② 로컬 프로그램이 로컬 파일을 읽는 것(컴퓨터 밖으로 안 나감, 도구의 핵심 가치). ②는 유지하되 자동 내용 읽기를 최소화: **존재만 자동, 내용은 클릭 시.**
- **대안:** 전체 자동 스캔(편하나 내용 자동 읽기) / 수동 붙여넣기(통제 최대지만 자동 발견·drift 상실) → 기각.
- **출처:** 인터뷰 (2026-06-30)

## D10. agent 포맷 검증 결과 — plan 가정 정정

- **결정:** agent 트랙은 공식 문서로 확인한 실제 포맷을 따른다. Claude·Gemini = `.md`+frontmatter(본문=system_prompt). **Codex = `.toml`, 시스템 프롬프트 = `developer_instructions`, 에이전트별 `tools` 필드 없음(→ Codex로 변환 시 tools는 "손실"로 표시).** Codex agents는 `.toml`만 인정(비표준 `.md`는 스킵).
- **맥락:** plan 초안은 Codex를 `instructions` 필드로 적었고 tools를 세 도구 모두 매핑 가능하다고 가정했으나, 공식 문서 확인 결과 다름. 실기기 `~/.codex/agents/`엔 `.md` 파일만 있었는데 이는 Codex 네이티브가 아님(스펙은 `.toml`).
- **근거·인용:** `docs/verification/agent-format-verification.md` (URL + 확인 문장 기록).
- **출처:** 0번 검증 (2026-06-30)

## D11. skill 트랙 위치·정책 (0번 검증 + 정정)

- **결정:** skill 스캔 위치 —
  - claude: `~/.claude/skills` + 플러그인(`~/.claude/plugins/**/skills`, 재귀)
  - codex: `~/.agents/skills`(공식·공유) + `~/.codex/skills`(비공식, 실재)
  - gemini: `~/.gemini/skills` + `~/.agents/skills`(공유·우선)
  - **`.agents/skills`는 "에이전트 소속"이 아니라 여러 도구가 공유하는 cross-tool 표준 위치**(공식 문서 확인). "공유" 출처로 표시.
  - **에이전트 전용 skill 폴더 개념은 없음** → skill은 디렉토리 기준으로만 스캔. (에이전트↔skill 연결 보기는 후속 별도 기능)
  - **비공식 위치(`~/.codex/skills`)의 skill은 전부 읽되, "비공식"으로 사용자에게 알림/제안**(공식 위치로 이동 제안 등).
- **맥락:** 처음엔 "`~/.codex/skills`=전역, `.agents/skills`=에이전트 소속"으로 추정했으나, 공식 문서 확인 결과 반대에 가까움(`.agents/skills`=공유 표준, `~/.codex/skills`=문서에 없음/레거시 가능).
- **근거·인용:** `docs/verification/skill-format-verification.md` + Codex skills 공식 문서.
- **출처:** 0번 검증 + 인터뷰 (2026-06-30)

## D12. 실제/더미 데이터 토글

- **결정:** 대시보드 상단 토글로 데이터 소스를 **dummy(fixtures) ↔ real(`~/.claude` 등)** 전환. **기본은 dummy(안전)**. canonical·backups도 모드별로 분리 — dummy는 `.dummy/canonical`, real은 `canonical/`. 실제 데이터는 **사용자가 토글로 직접** 읽고, AI(개발)는 더미 모드로만 검증한다(실제 데이터를 대화로 추출하지 않음).
- **맥락:** 사용자가 "실제 데이터를 AI와의 대화가 아니라 완성된 로컬 코드로 가져오고 싶다 + 상단에 실제/더미 토글" 요청. [[D9]](개발 더미 전용)와 일관.
- **구현:** `cli/mode.js`(상태) + `cli/paths.js`(모드별 경로) + 서버 `/mode` 엔드포인트 + 대시보드 상단 라디오 토글(+실제 모드 경고).
- **출처:** 인터뷰 (2026-06-30)

## D13. [인시던트] 실제 데이터 canonical 유입 + 실제 모드 옵트인 가드

- **인시던트:** Phase 2 telemetry 서브에이전트가 "더미 모드만" 지시를 어기고 **실제 모드 지시문 pull을 `canonical/`(+git 히스토리)에 실행** → 실제 `~/.claude/CLAUDE.md`·`~/.codex/AGENTS.md`·`~/.gemini/GEMINI.md` 내용이 canonical에 유입. **로컬 전용(원격 push 없음, [[D5]])이라 외부 유출은 없었으나** [[D9]]/[[D12]](개발은 더미만, 실제는 사용자만) 원칙 위배. 해당 서브에이전트가 실제 지시문을 자기 컨텍스트로 읽음(되돌릴 수 없음). 메인 컨텍스트엔 헤더 1줄만 유입.
- **조치 (즉시):**
  1. `canonical/` **git 재초기화** — 실제 데이터 + 히스토리 완전 제거(.gitignore만 남김). 테스트 잔여 `backups/`도 제거.
  2. **코드 가드** — 실제 모드는 `AAD_ALLOW_REAL=1` 옵트인 없이는 켜지지 않음(`cli/mode.js`의 `setMode('real')`·서버 `POST /mode` 차단, 대시보드 라디오 잠금). 서브에이전트·헤드리스·실수의 실제 데이터 접근을 **코드로 차단**.
  3. 더미 전용 canonical(`.dummy/canonical`)과 실제 canonical(`canonical/`) 분리 유지.
- **재발 방지 원칙:** 서브에이전트에게 보안은 **지시(프롬프트)만으로 의존하지 말고 코드 가드로 강제**한다. 실제 데이터를 다루는 동작은 항상 명시적 옵트인 뒤에만.
- **출처:** 인시던트 + 인터뷰 (2026-06-30)

## D14. 주기 점검 없음 + 마지막 동기화/점검 시각 표시

- **결정:** 자동 스케줄(cron/launchd) 점검은 **두지 않는다.** review·push는 **사용자가 누를 때만** 실행. 대신 **마지막 push(도구별)·마지막 review 시각**을 기록·표시(`cli/synclog.js` + `/synclog` + 대시보드 상단 줄). **B 신호(실제 호출 로그 파싱)는 이번에 구현하지 않음**(목적만 설명; 실제 로그가 필요해 추후 실제 모드에서). A 신호(git churn+mtime)만으로 사용량/노후화 동작.
- **맥락:** 사용자가 "주기 점검 없이, sync할 때만 동기화되고 마지막 시각만 표시" 요청. B는 "무엇을 위한 것"만 확인하고 구현은 보류.
- **출처:** 인터뷰 (2026-06-30)

## D15. 빌드 형태 = 브라우저 방식(현재 구조) 확장

- **결정:** 신규 3기능(분류·스토어·플레이그라운드)도 **현재 구조(로컬 CLI 코어 + 127.0.0.1 서버 + 브라우저 대시보드)를 확장**해 만든다. **Claude Code 플러그인으로 만들지 않는다.** 네이티브 앱(Tauri/Electron)은 나중에 "배포가 필요할 때" 같은 UI를 감싸는 선택지로 남긴다.
- **맥락:** 플러그인은 ① Claude Code 안에서만 동작해 핵심 가치인 "3개 도구 모두 적용"이 불가하고 ② 스토어/플레이그라운드 같은 독자 UI·실제 실행 샌드박스를 띄울 자리가 없다. 현재 구조는 이미 '로컬 앱'이라 3기능을 그대로 얹을 수 있고, 나중에 Tauri로 감싸도 전환 비용이 작다.
- **대안:** Claude Code 플러그인(기각 — 도구 종속·UI 불가) / 지금 네이티브 앱 전환(기각 — 기능도 없는데 패키징·서명·배포부터 떠안음).
- **출처:** 인터뷰 (2026-06-30). 유사 OSS 2종이 모두 네이티브(Tauri)인 점은 [[D19]] 참조.

## D26. UX 원칙 — "이해 안 되는 화면 금지" + "우아한 실패"

- **결정:** 대시보드 UI의 두 원칙을 확정한다.
  1. **이해 안 되는 화면 금지** — 사용자가 화면을 봤을 때 뜻을 모르는 요소가 없어야 한다. 최대한 간단한 용어로 기술하고, 전문어(churn/stale/drift/LRU 등)는 한글 라벨 또는 설명 툴팁 없이 노출하지 않는다. 같은 대상은 같은 단어로(도구/모델 혼용 금지 — "도구"로 통일).
  2. **우아한 실패** — 오류가 나도 화면이 죽거나(빈 화면·멈춤) 크래시하지 않는다. "이슈가 생겼다"는 표시(토스트/인라인 안내 + 재시도)를 하고 나머지 화면은 계속 동작한다.

## D27. UI 주인공 = 도구 간 동기화 매트릭스 (중앙 저장소는 뒤편 백업으로)

- **결정:** 화면의 중심 개념을 바꾼다. **중앙 저장소(canonical)는 사용자에게 중요하지 않다 — 백업/동기화 경유지로서 뒤편에 존재**할 뿐이다. 사용자에게 중요한 것은 **"홈(~)과 각 프로젝트에 있는 스킬·에이전트가 서로 동기화되어 있는가"**. 따라서:
  1. 스킬·에이전트 뷰 = **단일 매트릭스 표**: `경로 | 이름 | 태그 | claude | codex | gemini | 동기화`. 중앙 저장소 상태 표시는 주 화면에서 제거.
  2. 동기화 컬럼 상태(간단 용어): 동기화됨(모두 동일) / 불일치(내용 다름) / 일부 없음(있는 것끼리 동일) / 단독. [동기화] 버튼 = 기존 sync(기준 도구 선택→일괄 반영).
  3. **등록 없이 자동 발견**: 첫 실행에 1회 스캔(기존 결정 유지)하되 발견된 프로젝트를 **자동 등록**(사용자의 '등록' 단계 제거) + [다시 찾기] 수동 재스캔.
  4. 표의 동기화 상태 계산을 위해 로컬 파일 내용 해시 비교를 표 로드 시 수행 — D9의 "내용은 클릭 시"는 **AI 컨텍스트 유출 방지가 목적이었고(D24로 해소)**, 로컬 앱이 로컬 파일을 읽는 것은 원래 허용 범위.
  5. canonical은 sync 파이프라인(기준→canonical→각 도구)과 백업/복구 UI의 데이터로만 쓰임 — "중앙저장소는 백업으로써 존재".
- **맥락:** 사용자: "중앙저장소라는 게 중요한 게 아니고 백업으로써 존재… 현재 루트/프로젝트의 스킬·에이전트가 서로 sync 되어 있는지가 중요해. 등록 안 해도 모든 경로의 스킬을 볼 수 있게." 초기 plan1의 "행=항목, 열=도구 존재/drift 매트릭스" 설계로의 회귀이기도 함.
- **출처:** 인터뷰 (2026-07-02)

## D36. 프로젝트 관리 — 사라진 프로젝트 정리(prune)

- **결정:** 프로젝트 레지스트리에 등록됐지만 **디스크에서 사라진(root 미존재)** 프로젝트를 정리하는 기능을 추가한다.
  - `GET /projects`의 각 항목에 `exists`(stat만 — 내용 안 읽음) 플래그 노출.
  - `POST /projects/prune`(= `core.projectsPrune` → `projects.prune`): root가 없는 항목만 제거, `{removed:[...], remaining:N}` 반환. `aad projects prune` CLI parity(+`aad projects ls`에 `[없음]` 표시).
  - UI: 프로젝트 관리 모달에서 없는 프로젝트 행은 앰버 톤 + "없음" 배지, 없는 게 있을 때만 [없는 프로젝트 N개 정리] 버튼 노출. 정리 후 매트릭스 자동 새로고침.
- **초기화(reset)와 구분:** reset은 목록을 비우고 처음부터 재스캔(전량 재등록), prune은 **사라진 것만** 제거하고 나머지 등록은 그대로 둔다(수동 추가분 보존).
- **보안:** 존재 확인은 `fs.statSync`만 사용(디렉토리 여부) — 파일을 열지 않는다. 격리 스모크로 실존/미존재 혼합 레지스트리에서 미존재만 제거됨을 확인.

## D37. 더미/데모 모드 전면 제거 — 항상 실제 파일 기준

- **결정:** 더미(데모) 모드와 실제-모드 옵트인 가드를 코드째 제거한다. 앱은 **항상 실제 내 파일 기준**으로 동작한다.
  1. 삭제: `cli/mode.js`(AAD_ALLOW_REAL/AAD_MODE 게이트), `fixtures/`(가짜 모델 홈 샘플), `.dummy/`(더미 작업 공간), `GET/POST /mode` 라우트, 대시보드 더미 안내 배너, `npm run demo`/`demo:serve` 스크립트, launch.json `aad-dummy`.
  2. `paths.js`는 분기 없이 실제 경로만: 모델 홈 = `~/.claude`·`~/.codex`·`~/.agents`, canonical = `<repo>/canonical`(플러그인은 DATA), 백업 = `<repo>/backups`(gitignore 추가).
