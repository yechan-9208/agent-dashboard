# Claude Design 요청 프롬프트 — AI 작업 지침서 관리 대시보드 발표 자료

아래 내용을 바탕으로 **5~7분 발표용 PPT/슬라이드 덱**을 디자인해줘.  
대상 청중은 개발자가 아니라 **사장님, 부사장님 등 비개발자 임원**이 포함되어 있으므로, 어려운 개발 용어는 최대한 피하고 쉽게 이해되는 비유와 화면 구성을 사용해줘.

---

## 1. 발표 제목

**흩어진 AI 작업 지침서를 한 화면에서 확인하기**

### 부제
AI를 활용해 만든 로컬 지침서 관리 대시보드

### 발표자
안드로이드팀 원예찬 사원

---

## 2. 발표 배경

회사 AI 경진대회 주제는 **“AI로 내 일을 바꾼다”**이다.

발표자는 안드로이드 개발자로서 업무에서 AI를 자주 사용한다.

AI 사용 예시:

- 코딩 보조
- 번역
- 문서 작성
- 파일 찾기
- 오류 내용 정리
- 작업 내용 요약

발표의 핵심은 “AI가 무언가를 자동으로 다 해준다”가 아니라, **AI를 더 잘 쓰기 위한 작업 지침서를 만들고 관리하는 도구를 만들었다**는 것이다.

---

## 3. 반드시 전달해야 하는 핵심 메시지

발표 전체를 관통하는 문장은 다음이다.

> **AI를 잘 쓰려면 좋은 질문뿐 아니라, 좋은 지침을 저장하고 관리하는 체계도 필요하다.**

또 다른 핵심 문장:

> **동기화보다 먼저 필요한 것은, 내가 가진 AI 작업 지침서를 한눈에 보는 것이었다.**

결론 문장:

> **AI 지침서도 업무 자산이기 때문에, 만들고 끝나는 것이 아니라 관리가 필요하다.**

---

## 4. 용어 설명 — 비개발자용 표현 사용

슬라이드에는 가능한 한 아래 쉬운 표현을 사용해줘.

| 기술 용어 | 발표용 쉬운 표현 |
|---|---|
| skill | AI 작업 지침서 |
| instruction | 기본 작업 지침 |
| sync | 필요할 때 맞춰 쓰기 |
| diff | 바뀌는 부분 미리보기 |
| backup | 되돌릴 수 있게 저장 |
| local | 내 컴퓨터 안에서 실행 |
| CLI | 개발자용 AI 도구 |
| scan | 찾아보기 |
| dashboard | 현황판 / 관리 화면 |

피해야 할 단어 또는 최소화할 단어:

- agent
- playground
- Gemini
- TOML
- frontmatter
- canonical
- environment variable
- 127.0.0.1
- filesystem scan
- API route

필요하면 발표자가 구두로 짧게 설명할 수 있으므로, 슬라이드에는 가급적 넣지 말아줘.

---

## 5. 디자인 방향

청중이 비개발자이므로, 기술 대시보드처럼 복잡하게 만들지 말고 **문제 → 비유 → 해결 → 결과** 흐름이 바로 보이게 디자인해줘.

### 톤

- 깔끔한 실무 발표형
- 임원 보고용으로 너무 장난스럽지 않게
- 하지만 딱딱한 기술 문서처럼 보이지 않게
- “업무 매뉴얼 / 지침서 / 현황판” 비유를 중심으로

### 시각 스타일

- 밝은 배경: white / light gray 계열
- 강조색: deep blue 또는 teal 계열 1개
- 과한 그라데이션, 유리 효과, AI 느낌의 과장된 장식은 피하기
- 한 슬라이드에 문장 3~5개 이하
- 텍스트보다 표, 흐름도, 폴더 그림, 비교 카드 중심
- 아이콘은 필요한 만큼만 사용

### 추천 레이아웃

- 좌우 비교
- 3단계 흐름도
- 폴더 두 개가 흩어져 있고 가운데 물음표가 있는 그림
- 마지막에는 한 화면에서 정리된 현황판 형태

# Slide 2. AI는 이미 업무 보조 도구

## 제목
AI는 이미 제 업무 보조 도구입니다

## 본문
- 코딩 보조
- 번역
- 문서 작성
- 파일 찾기
- 오류 내용 정리
- 작업 내용 요약

## 디자인 요청
- 6개 항목을 아이콘 카드로 표현
- 각 카드에는 한 단어 또는 짧은 문구만 넣기
- “AI를 많이 사용한다”는 사실을 직관적으로 보여주기

## 발표자가 말할 내용
AI 경진대회 주제가 “AI로 내 일을 바꾼다”였습니다.  
실제로 저는 업무에서 AI를 많이 사용합니다.  
안드로이드 개발자로서 코딩뿐만 아니라, 번역, 문서 작성, 파일 찾기, 오류 내용 정리 등 여러 작업을 AI에게 지시해서 도움을 받고 있습니다.  
그런데 AI를 사용하면서 느낀 점은, AI에게 얼마나 자세하게 지시하느냐에 따라 결과가 크게 달라진다는 점입니다.

# Slide 3. 자세한 지시가 결과를 바꾼다

## 제목
AI는 지시가 자세할수록 결과가 좋아집니다

## 슬라이드 구성
좌우 비교 카드로 구성.

### 왼쪽 카드 — 짧은 지시

**입력**
```text
이 오류 내용 정리해줘.
```

**결과**
```text
오류가 발생했습니다.
해당 부분을 확인해야 합니다.
```

### 오른쪽 카드 — 자세한 지시

**입력**
```text
개발팀 공유용으로 정리해줘.
확인된 사실과 추정을 분리해줘.
현상 / 근거 / 원인 후보 / 다음 액션으로 나눠줘.
개인정보나 고객정보는 포함하지 마.
```

**결과**
```text
현상: 로그인 후 앱 종료
근거: 오류 로그 확인
원인 후보: 값이 비어 있을 가능성
다음 액션: 재현 조건과 예외 처리 확인
```

## 디자인 요청
- 왼쪽은 회색/단순한 느낌
- 오른쪽은 강조색을 사용해 “업무에 맞는 결과” 느낌
- 가운데 화살표 또는 “같은 요청도 지시에 따라 달라짐” 문구
- 실제 사내 정보처럼 보이지 않게 더미 예시임을 작게 표시

## 발표자가 말할 내용
같은 내용을 AI에게 요청하더라도 지시 방식에 따라 결과가 달라집니다.  
단순히 “이 오류 정리해줘”라고 하면 AI는 간단한 요약만 해줍니다.  
하지만 개발팀 공유용인지, 어떤 형식으로 정리해야 하는지, 확인된 사실과 추정을 분리해야 하는지까지 알려주면 훨씬 업무에 맞는 결과가 나옵니다.

# Slide 4. 좋은 지시는 지침서로 저장한다

## 제목
좋은 지시는 일회용으로 쓰기 아깝습니다

## 본문 흐름
```text
좋은 지시
  ↓
문서로 저장
  ↓
다음 작업에서 재사용
  ↓
결과 품질을 일정하게 유지
```

## 보조 문구
기술적으로는 이런 문서를 `skill`이라고 부르기도 하지만, 발표에서는 쉽게 **AI 작업 지침서**라고 표현합니다.

## 디자인 요청
- “업무 매뉴얼” 비유가 느껴지게 문서/책자 형태 아이콘 사용
- 흐름도 형태로 단순하게 표현
- `skill`이라는 단어는 작게만 표시하고, 핵심은 “AI 작업 지침서”로 강조

## 발표자가 말할 내용
이런 자세한 지시사항을 매번 기억해서 다시 쓰기는 어렵습니다.  
그래서 자주 쓰는 지시는 문서로 저장해두고, 다음에 AI에게 “이 지침서를 기준으로 작업해줘”라고 하면 이전과 비슷한 품질의 결과를 기대할 수 있습니다.  
사람에게 업무 매뉴얼이 필요하듯이, AI에게도 업무 지침서가 필요하다고 생각하시면 됩니다.

# Slide 5. 웹 AI와 개발자용 AI 도구는 다르다

## 제목
웹 AI와 개발자용 AI 도구는 사용 방식이 다릅니다

## 표
| 웹 AI | 개발자용 AI 도구 |
|---|---|
| 웹사이트에서 질문 | 내 컴퓨터 작업 폴더에서 실행 |
| 내용을 복사해서 입력 | 파일과 폴더를 직접 참고 |
| 일회성 질문에 적합 | 반복 개발 업무에 적합 |
| 지침을 매번 입력 | 지침서를 파일로 저장해 재사용 |

## 강조 문구
이 도구의 장점은 **내 컴퓨터 안에 실제로 어떤 지침서 파일이 있는지 직접 찾아서 보여줄 수 있다는 점**입니다.

## 디자인 요청
- 왼쪽: 웹 브라우저/채팅창 그림
- 오른쪽: 노트북 + 프로젝트 폴더 그림
- 오른쪽에 강조색 사용
- 로컬 실행을 보안 만능처럼 설명하지 말 것
- 핵심은 “관리 대상이 내 컴퓨터 안의 실제 파일/폴더”라는 점

## 발표자가 말할 내용
보통 AI라고 하면 ChatGPT 웹사이트에 들어가서 질문하는 모습을 많이 떠올리실 것 같습니다.  
저도 번역이나 문서 작성은 그런 방식으로 사용합니다.  
그런데 개발 업무에서는 AI를 웹 채팅창에서만 쓰지 않습니다.  
Claude Code나 Codex 같은 개발자용 AI 도구는 제 컴퓨터의 개발 폴더 안에서 실행되면서, 프로젝트 파일을 보고 필요한 파일을 찾고, 제가 만들어둔 작업 지침서를 참고하면서 도와줍니다.

# Slide 6. 문제: 지침서가 흩어진다

## 제목
문제는 “내가 뭘 가지고 있는지 모른다”는 것

## 본문
- 지침서가 하나둘 쌓임
- 어떤 것은 Claude 쪽에 있음
- 어떤 것은 Codex 쪽에 있음
- 프로젝트별로 따로 생김
- 같은 이름인데 내용이 달라질 수 있음
- 시간이 지나면 찾기 어려움

## 강조 문구
**동기화보다 먼저 필요한 것은, 내가 가진 AI 작업 지침서를 한눈에 보는 것**

## 디자인 요청
- 두 개의 폴더를 시각화:
  - Claude 쪽 폴더
  - Codex 쪽 폴더
- 각 폴더 안에 다른 이름의 문서들이 들어있는 형태
- 가운데에 물음표 또는 “어디에 뭐가 있지?” 표시
- 비개발자도 “문서가 흩어져 있다”는 점을 바로 이해하게

## 발표자가 말할 내용
문제는 이런 지침서가 하나둘 쌓이다 보면, 제 컴퓨터 어디에 어떤 지침서가 있는지 알기 어려워진다는 점입니다.  
특히 Claude Code와 Codex처럼 서로 다른 AI 코딩 도구를 같이 쓰면, 각 도구가 지침서를 저장하는 위치가 다릅니다.  
그래서 저는 먼저 내 컴퓨터에 어떤 AI 작업 지침서가 있는지 한눈에 확인하는 도구가 필요하다고 생각했습니다.

# Slide 7. 해결: 로컬 지침서 현황판

## 제목
흩어진 지침서를 한 화면에 보여주기

## 도구가 하는 일
1. 내 컴퓨터에서 지침서 파일 찾기
2. 어떤 지침서가 있는지 목록화
3. Claude / Codex 중 어디에 있는지 표시
4. 한쪽에만 있는 지침서 확인
5. 같은 이름인데 내용이 다른 지침서 확인
6. 필요할 때만 미리보기 후 맞춰 쓰기

## 실제 결과 예시
```text
전체 지침서: 8개
서로 같은 지침서: 3개
일부 도구에 없는 지침서: 1개
한쪽에만 있는 지침서: 4개
```

## 표 예시
| 지침서 | Claude | Codex | 상태 |
|---|---|---|---|
| code-review | 있음 | 없음 | 일부 없음 |
| docx | 있음 | 있음 | 동일 |
| shared-format | 없음 | 있음 | 한쪽만 있음 |

## 디자인 요청
- 실제 대시보드처럼 보이는 단순한 표/현황판 디자인
- 숫자 4개는 상단 요약 카드로 보여주기
- 표는 3행 정도만, 복잡하게 만들지 말 것
- “현황판” 느낌 강조

## 발표자가 말할 내용
제가 만든 도구는 내 컴퓨터에서 AI 작업 지침서 파일을 찾고, 어떤 지침서가 있는지 목록으로 보여줍니다.  
그리고 그 지침서가 Claude Code에 있는지, Codex에 있는지 확인할 수 있게 합니다.  
가장 중요한 목적은 현황을 한눈에 보는 것입니다.  
이후 필요할 때는 변경 내용을 미리 보여주고, 사람이 확인한 뒤 맞춰 쓸 수 있게 했습니다.

# Slide 8. AI 활용 방식: 단계별로 지시하고 검토했다

## 제목
AI에게 한 번에 맡기지 않고 단계별로 진행했습니다

## 흐름
```text
Claude
기획 
  ↓
작업 분리, 결정사항 기록
  ↓

Claude code
개발 기획
  ↓
개발 에이전트
  ↓
검증 에이전트
  ↓
직접 UX 판단
  ↓
수정 요청
  ↓
이거는 반복하도록


Claude Design
디자인
  
```

## 보조 문구
AI는 단순 작성자가 아니라, 문제 정리·구현·검증을 돕는 협업자로 활용했다.

## 디자인 요청
- Claude, Claude code, Claude design 각 단계는 배경색 다르게 구분해서 표시
- 세로 또는 가로 프로세스 다이어그램
- 각 단계는 짧은 단어 중심
- “반복”이 느껴지도록 마지막에서 다시 수정 요청으로 돌아가는 순환 화살표가 있어도 좋음

## 발표자가 말할 내용
이번 프로젝트에서 AI는 단순히 코드를 작성하는 데만 사용하지 않았습니다.  
문제를 정리하고, 어떤 파일과 구조를 봐야 하는지 조사하고, Claude와 Codex의 공식 문서를 확인하는 데 활용했습니다.  
AI에게 한 번에 큰 결과를 맡긴 것이 아니라, 작업을 나누고, 결과를 확인하고, 마음에 들지 않는 부분은 다시 수정 요청하는 방식으로 진행했습니다.

그래서 불필요하게 복잡한 기능은 줄이고, 지침서 현황 확인에 집중했습니다.

# Slide 10. 결론

## 제목
AI 지침서도 업무 자산입니다

## 본문
- AI는 자세한 지시가 중요하다
- 좋은 지시는 저장해서 재사용해야 한다
- 지침서가 쌓이면 관리가 필요하다
- 내 컴퓨터 안의 지침서를 한눈에 보는 현황판을 만들었다
- AI를 활용해 설계·구현·검증했다

## 마지막 강조 문장
**AI를 잘 쓰려면, 좋은 질문뿐 아니라 좋은 지침을 관리하는 체계도 필요합니다.**

## 디자인 요청
- 마지막 슬라이드는 메시지 중심
- 이전 슬라이드보다 여백을 넓게 사용
- 하단에 작은 흐름: 좋은 지시 → 지침서 → 관리 → 재사용

## 발표자가 말할 내용
이 도구는 모든 사람에게 필요한 범용 앱은 아닙니다.  
하지만 AI를 단순 질문용이 아니라 업무 도구로 깊게 쓰기 시작하면, 좋은 지침을 만들고 관리하는 문제가 생깁니다.  
이번 프로젝트를 통해 느낀 점은, AI를 잘 쓰려면 좋은 질문을 하는 것뿐만 아니라, 좋은 지침을 저장하고 관리하는 체계도 필요하다는 점입니다.  
저는 AI를 활용해, 제 컴퓨터에 흩어진 AI 작업 지침서를 한눈에 확인하고 필요할 때 맞춰 쓸 수 있는 도구를 만들어봤습니다.

## 8. 예상 질문 대비용 백업 문구

PPT 본문에는 다 넣지 않아도 되지만, 발표자가 답변할 수 있도록 발표 노트나 부록에 아래 내용을 포함해도 좋다.

### Q. 왜 로컬에서 만들어야 하나요?
관리하려는 대상이 웹에 있는 자료가 아니라 제 컴퓨터 안에 실제 파일로 저장된 AI 작업 지침서이기 때문입니다. 개발자용 AI 도구는 프로젝트 폴더 안에서 실행되면서 파일과 지침서를 참고하기 때문에, 로컬 파일을 직접 찾아서 보여주는 방식이 필요했습니다.

### Q. 이 도구의 핵심 기능은 뭔가요?
동기화보다 먼저, 제 컴퓨터에 어떤 AI 작업 지침서가 있는지 한눈에 확인하는 것입니다. 어떤 지침서가 어느 AI 도구에 있고, 한쪽에만 있는지, 서로 다른지를 보는 현황판 역할이 핵심입니다.

### Q. 왜 동기화 기능도 있나요?
토큰 부족 또는 새 모델이 계속 나오면서 작업에 따라 Claude Code와 Codex를 번갈아 쓰는 경우가 생깁니다. 그때 한쪽에서 잘 쓰던 지침서를 다른 쪽에서도 쓰고 싶을 수 있기 때문에, 필요한 경우에만 사람이 확인하고 맞춰주는 기능을 넣었습니다.

### Q. 자동으로 다 맞춰주나요?
아닙니다. 자동 적용이 아니라, 목록 확인 → 변경 내용 미리보기 → 사람 확인 → 백업 후 적용 흐름입니다. 사람이 최종 판단합니다.

### Q. 개발자가 아닌 사람에게도 필요한가요?
현재 구현은 개발자용 AI 도구를 대상으로 합니다. 다만 여러 AI 도구에 업무 매뉴얼, 보고서 양식, 검토 기준이 흩어지는 문제는 다른 직무에도 생길 수 있기 때문에, 개념적으로는 확장 가능하다고 봅니다.

### Q. 기능 범위를 줄인 이유는 뭔가요?
처음에는 여러 기능을 검토했지만, 실제 문제의 핵심은 “내가 만든 AI 지침서들이 어디에 있고 어떤 상태인지 모른다”는 것이었습니다. 그래서 핵심 문제에 집중하기 위해 범위를 줄였습니다.
