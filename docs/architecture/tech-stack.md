# 기술 스택과 선택 근거 (Tech Stack & Rationale)

AAD를 **어떤 언어·도구로, 왜 그렇게** 만들었는지 정리한 문서다.
각 선택의 근거는 개발 중 사용자와의 대화(인용)와 결정 로그([`decision.md`](../decision/decision.md) D번호)에 둔다.
모듈이 어떻게 나뉘는지는 [`structure.md`](structure.md), 보안 규칙은 [`../security/security.md`](../security/security.md) 참고.

## 한눈에 보기

| 외부 의존성 | **2개뿐** — `gray-matter` · `@iarna/toml` | 로컬 신뢰 도구의 공급망 최소화. 나머지는 전부 Node 내장 모듈 |

## 1. 언어·런타임 — Node.js (CommonJS)

- **선택:** Node.js 단일 런타임. 트랜스파일 없는 CommonJS(`require`).
- **근거:**
  - 이 도구의 일은 전부 **로컬 파일시스템 조작**(스캔·읽기·diff·쓰기·백업)이다. Node 내장
    `fs`/`path`/`http`/`crypto`만으로 CLI(`bin/aad.js`)·서버(`server/serve.js`)·정적 프런트
    서빙을 모두 처리할 수 있어 계층마다 다른 기술을 둘 이유가 없었다.
  - Claude Code·Codex 사용자는 이미 Node가 깔린 환경이 대부분 — 설치 마찰 최소화. 이는 배포를
    플러그인으로 정하면서(D34) 더 중요해졌다: `/aad:serve` 한 번으로 `npm install`까지 자동.
  - 초기 기획(1단계 지시서)부터 "macOS 로컬 환경에서 동작"이 전제였고, TypeScript/빌드 단계는
    로컬 단일 사용자 도구에 과했다.

## 2. 의존성 정책 — 2개만

- **선택:** `gray-matter`(마크다운 frontmatter 파싱) + `@iarna/toml`(Codex `.toml` 에이전트 파싱). 끝.
- **근거:**
  - 이 앱은 사용자의 **개인 설정 파일 전체를 스캔할 권한**을 가진 로컬 신뢰 도구다. 의존성
    하나하나가 공급망 리스크라서 "외부 의존성 없이 Node 내장 모듈만"을 기본 원칙으로 시작했고
    (package.json 주석에 명시), 파서 2개만 예외로 들였다 — frontmatter와 TOML은 직접 구현할
    가치가 없는 검증된 포맷 파서이기 때문.
  - diff(LCS), 검색 랭킹, 유사도(jaccard), HTTP 라우팅은 전부 직접 구현(`cli/diff.js` 등) —
    수백 줄 수준이라 라이브러리보다 통제가 쉽다.

## 3. 서버 — 내장 `http` + 127.0.0.1 전용

- **선택:** Express류 프레임워크 없이 Node 내장 `http`. **`127.0.0.1`에만 바인딩**, 서버는
  `cli/core.js`를 호출만 하는 "얇은 다리"(로직 중복 0).
- **근거:**
  - **로컬 전용이 제품의 정체성이자 보안 모델이다.** 사용자: *"해당 프로젝트는 외부와 통신이
    되지 않으니 사용자만 볼 수 있기 때문에"* — 이 전제로 개인정보(PI) 게이트까지 제거했다(D33).
    바인딩을 코드로 127.0.0.1에 고정했기에 성립하는 논리다.
  - 라우트가 30여 개뿐이고 전부 `core` 함수 1:1 매핑이라 프레임워크가 주는 것이 없다.
    **단일 로직 소스** 원칙(CLI와 웹이 같은 `core.js`를 씀)이 구조의 핵심이고, 서버는 그 원칙을
    지키는 배관일 뿐이다.

## 4. 프런트엔드 — 바닐라 JS SPA (빌드 없음)

- **선택:** React/Vue 없이 `index.html` + `app.js` + `design-system.css` 3파일. 해시 라우터 +
  `fetch` + **단일 `document` 클릭 위임**(모든 버튼은 `data-*` 속성 분기).
- **근거:**
  - 빌드 단계가 없어야 "clone → `npm run serve`"로 끝난다. 번들러·개발서버·프레임워크 버전
    관리가 통째로 사라진다. 화면이 4뷰뿐이라 상태 관리 라이브러리도 불필요.
  - 단일 클릭 위임은 동적으로 다시 그리는 표·모달에 리스너를 매번 다시 붙이는 버그를 원천
    차단한다(렌더 함수는 HTML 문자열만 만들고, 동작은 `data-*` 계약으로 한곳에서 처리).
  - UI 방향도 사용자 결정: 처음부터 *"플러그인은 독자 UI 불가라 브라우저 방식"*(D15)이었고,
    반응형·다크모드 제거·용어("모델") 등은 UX 대개편 때 확정했다(D28~D30).

## 5. 데이터 저장 — 파일 + JSON, 백업은 자체 git

- **선택:** DB 없음. 상태는 JSON 파일(`projects.json`, 사용빈도 커서·이벤트), 백업은
  `canonical/`(중앙 저장소)이 **자체 git 저장소**로 이력을 가진다. 덮어쓰기 전 타임스탬프 백업.
- **근거:**
  - 다루는 대상 자체가 마크다운/TOML **파일**이다. "파일을 파일로 관리"가 가장 투명하고,
    사용자가 언제든 텍스트 에디터로 직접 열어 검증할 수 있다(로컬 신뢰 도구의 요건).
  - 이력·되돌리기는 git이 이미 완벽하게 한다. 단, canonical git은 **자기 자신이 저장소 루트일
    때만 커밋**하도록 가드(상위 repo 오염 사고를 겪고 코드로 차단).
  - 모든 쓰기는 **미리보기(diff) → 명시적 적용** 2단계 + 자동 백업 — 처음부터 불변 원칙.

## 6. 2모델 체제 — Claude Code · Codex

- **선택:** 지원 대상을 Claude Code와 OpenAI Codex 2개로 고정. Gemini는 코드 수준에서 제거.
- **근거:** 초기에는 3모델(Claude·Codex·Gemini)로 시작했고 Gemini 공식 지원 여부까지
  재검증했지만, 실사용에서 Gemini를 쓰지 않게 되자 사용자가 정리했다: *"제일 중요한 건 gemini는
  이제 지원하지 말자, 이제는 2개로 간단하게 가자"* (D31). 열 하나가 줄면 매트릭스·변환·UI 전부가
  단순해진다 — 유지 비용 대비 가치가 없는 축은 코드째 지운다.
- 같은 논리로 **에이전트 뷰도 숨겼다**(D32): Codex가 에이전트별 `tools` 필드를 지원하지 않아
  100% 왕복 동기화가 불가능한데, *"완전하지 않은 기능을 노출하지 않는다"* — 엔진은 남기고 UI만
  숨김(복원 조건은 `docs/plan/todo.md`).

## 7. 사용빈도 — 세션 기록의 호출 횟수(B신호)

- **선택:** 사용량을 파일 수정 시각(mtime)이 아니라 **Claude 세션 기록(`~/.claude/projects/*/*.jsonl`)의
  스킬 호출 횟수**로 센다(`cli/usagelog.js`). 이름·시각만 추출, 본문은 저장·표시하지 않는다.
- **근거:** mtime 근사치를 보여주자 사용자가 바로잡았다: *"아니, 얼마나 자주 사용했는지를 봐야
  한다고"* (D33). 파일이 바뀐 것과 스킬을 쓴 것은 다른 사실이다. Codex 세션 로그는 시크릿
  거부목록이라 읽지 않으며, 화면에 "Claude 세션 기준"을 명시한다.

## 8. 보안 — 프롬프트가 아니라 코드로

- **선택:** 보안 경계를 전부 **코드 가드**로 구현: `security.js`(시크릿 거부목록 — auth.json·
  *.sqlite·sessions/·.env 등은 절대 읽지 않음), 스캔 화이트리스트, pull 소스는 등록된 프로젝트만,
  모든 쓰기는 미리보기→적용 2단계 + 자동 백업. (초기에는 더미/실제 모드 가드(`AAD_ALLOW_REAL`
  옵트인)도 있었으나 "이제는 바로 실제 파일 기준"이라는 사용자 결정으로 D37에서 제거 — 안전은
  모드가 아니라 위 가드들이 담당한다.)
- **근거:** 개발 자체를 AI 서브에이전트에게 위임하는 방식이라, "하지 마라"는 프롬프트 지시만으로는
  부족하다 — **에이전트가 실수해도 코드가 막아야 한다**가 그룹개발 방법론의 보안 원칙이다.
  실제로 서브에이전트 검증도 격리 환경(`AAD_CANONICAL`/`AAD_BACKUPS` 오버라이드 + 임시 포트)에서만 수행했다.

## 9. 디자인 — Apple 스타일 디자인 시스템 v2

- **선택:** 사용자가 Claude 디자인으로 제작한 목업(`docs/design/AAD-Redesign-standalone.html`)을
  권위 소스로 전면 적용(D35). 시스템 폰트 스택(-apple-system → SF Pro, 폰트 임베드 없음),
  모델색(보라·청록)과 상태색(초록·앰버·회색) 완전 분리, 빌드 없이 CSS 변수 토큰으로 구현.
- **근거:** 사용자: *"이거 참고하여 현재 디자인 수정해줘, 왠만하면 이 디자인을 그대로 쓰는
  쪽으로."* 목업 안에 토큰·반응형·구현 체크리스트까지 정의돼 있어 디자인 결정을 코드 쪽에서
  재발명하지 않았다. 단 3가지는 대화로 조정: 지시문은 비교 전용 유지(쓰기 버튼 제외), 사용량
  3지표는 목업대로 복원, 반응형은 데스크톱까지만.

## 10. 배포 — Claude Code 플러그인

- **선택:** repo 자체를 플러그인화(`.claude-plugin/` + `commands/` + `scripts/`). `/aad:serve`가
  의존성 자동 설치 → 서버 실행 → 브라우저 열기까지 담당. 상태(백업·프로젝트 목록·node_modules)는
  업데이트에도 살아남는 `~/.claude/plugins/data/aad/`에 영속화.
- **근거:** 사용자: *"배포를 claude 플러그인으로 하려고 — 앱 자체를 플러그인으로 만들어서
  배포하게. 플러그인 명령어를 치면 서버 실행, 화면도 나올 수 있게"* (D34). 플러그인 캐시 경로는
  업데이트마다 바뀌므로(공식 스펙 검증) 상태를 캐시 밖 영속 폴더로 분리한 것이 설계의 핵심.

## 11. 개발 방식 — Claude Code + "그룹개발"

- **선택:** 개발 자체를 Claude Code로 진행하되, **그룹개발 방법론**([`../process/group-dev.md`](../process/group-dev.md))을
  적용: 계획 수립 → 사용자 확인 → 분리 가능한 작업을 서브에이전트에 병렬 위임 → 격리 환경
  검증 → 문서(결정 로그) 기록.
- **근거:**
  - 사용자 운영 원칙: 계획은 *"진행하기 전에 어떻게 나눠서 어떤 에이전트들이랑 작업할 건지
    확인을 받아"*, 커밋은 요청 시에만·의미 단위로 분리, 원격 push는 계정 확인 후에만.
  - 모든 큰 결정은 [`decision.md`](../decision/decision.md)에 D번호로 기록해 "왜"를 코드 밖에
    남긴다 — 이 문서(tech-stack)도 그 로그를 근거로 쓰였다.
  - 검증 도구: `node --check`(구문) + 격리 서버 스모크(`AAD_CANONICAL`/`AAD_BACKUPS`를 임시
    폴더로 돌린 뒤 임시 포트) + 브라우저 프리뷰(스크린샷 대조). 구현 전 외부 포맷은
    **"0번 검증"**(공식 문서 확인 보고서, [`../verification/`](../verification/))을 먼저 남긴다.
